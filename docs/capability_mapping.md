## 1. 目的

施設基準届出データ (厚生局届出受理名簿由来) を、医療機能の代理指標として **10カテゴリの capability** に集約する。これにより、以下が可能になる:

- **FacilityExplorer**: 施設ごとの cap値による医療機能の俯瞰
- **top_facilities**: 規模・実績スコアの構成要素として参照
- **Bridge supply**: ただし Bridge は別データソース (病床機能報告) を使用 — §9 参照

## 3. Matching Logic

```python
def classify(standard_name: str, taxonomy: dict) -> list[str]:
    """各施設基準名に対し、カテゴリごとの keyword list を部分一致で照合する。
    一致したカテゴリすべてに count を加算する。
    したがって、1つの施設基準が複数カテゴリに該当する場合がある。
    本仕様では multi-label を許容する。"""
    cats = []
    for cat, keywords in taxonomy.items():
        for kw in keywords:
            if kw and kw in standard_name:
                cats.append(cat)
                break  # 1カテゴリにつき1キーワードでマッチすれば充分
    return cats

# 1施設の cap値を計算
facility_cap = {c: 0 for c in taxonomy}
for st in facility["standards"]:
    for cat in classify(st["name"], taxonomy):
        facility_cap[cat] += 1
```

**設計判断**:
- 1キーワードでもマッチしたら該当カテゴリに加算 → 漏れを防ぐが false positive リスクあり (§9 参照)
- 同一基準が複数カテゴリにマッチする場合は **multi-label として両方に加算** (§8 参照)

## 6. Validation Result

### 6.1 taxonomy ロジック決定論性

✅ taxonomy logic は決定論的 (同一基準名から常に同じカテゴリリストを返す)

substring matching は **純粋関数** (副作用なし、入力→出力が1対1)。同じ施設基準名は常に同じカテゴリリストを返す。

### 6.2 全件サマリ (audit 実施結果)

| 指標 | 件数 | 割合 |
|---|---|---|
| ユニーク施設基準数 (raw) | 910 | 100% |
| matched (10カテゴリのいずれかに該当) | 338 | 37.1% |
| unmatched (どのカテゴリにも該当せず) | 572 | 62.9% |
| multi-match (2カテゴリ以上にマッチ) | 68 | 7.5% |

### 6.3 データソース別サンプル数

- `facility_standards.json` — 全国 6県サンプル (11,164 施設、ユニーク基準 910件)。本 audit はこれをベースに実施。
- `kijun_shards/*.json` — 47県完全 (cap値計算済)。47県分布の集計に使用。

両データソースは独立しており、**facility_standards = サンプル** / **kijun_shards = 全国集計値** という補完関係。`facility_standards` で抽出した taxonomy ロジックは、`kijun_shards` の cap 構造と **同じ keyword list** を共有するため、matching ロジックは 100% 一貫性を保つ。

## 7. Unmatched Standards

全 910 基準のうち 572 基準 (62.9%) は 10カテゴリのいずれにも該当しない。

### 7.1 設計上の判断

これは **意図的**:

> facility_taxonomy v1 がすべての施設基準を網羅する分類ではなく、医療機能探索に使いやすい高レベルカテゴリだけを抽出する設計であるため。  
> unmatched には、個別指導料、薬剤関連、特殊検査、細かな管理料などが含まれる可能性がある。  
> したがって `cap.*` は施設基準全体の総量ではなく、特定カテゴリに該当した届出の proxy である。

### 7.2 unmatched の主要グループ

実際の unmatched から抽出:

1. **薬剤_経口投与療法**: 「アスピリン経口投与療法」「イマチニブ経口投与」等の特定薬剤治療料
2. **検査_PCR/核酸検出**: 「ウイルス・細菌核酸多項目同時検出」「迅速診断 (PCR法)」等
3. **指導料・管理料**: 「ウイルス疾患指導料」「こころの連携指導料」等
4. **特殊加算**: 「せん妄ハイリスク患者ケア加算」「う蝕歯無痛的窩洞形成加算」等
5. **歯科関連**: 歯科診療所の届出基準 (大部分)

### 7.3 含意

`cap.*` の合計を「施設の届出総数」と読むことはできない。本指標は **特定カテゴリに該当した届出の proxy** に過ぎない。

> 同一施設基準が複数カテゴリに入るため、カテゴリ別cap合計を単純合算して「総cap」と読むことはできない。

## 9. Known Limitations (Devil's Advocate)

### 9.1 substring matching の False Positive リスク

audit (Phase 2B 実施) で発見した false positive 候補:

| カテゴリ | 例 | 評価 |
|---|---|---|
| `acute` | 「反復経頭蓋磁気刺激療法 うつ病（**急性期**...）」 | ⚠️ 精神療法だが、"急性期" キーワードで acute にマッチ |
| `acute` | 「急性期看護補助体制加算」 | ✅ 妥当 |
| `imaging` | 内視鏡下手術系 43件 (例: 内視鏡下甲状腺悪性腫瘍手術) | ⚠️ 外科手術だが "内視鏡" キーワードで imaging+surgery に重複 |
| `rehab` | 「歯科口腔リハビリテーション料2」 | ⚠️ 歯科系だが "リハビリ" キーワードでマッチ |
| `psychiatry` | 「ニコチン依存症管理料」 | ✅ 妥当 ("依存症" キーワード) |
| `rehab` | 「脳血管疾患等リハビリテーション料」 | ✅ 妥当 |

### 9.2 cap.oncology

> `cap.oncology` は、**がん診療連携拠点病院を意味しない**。  
> 外来化学療法、放射線治療、緩和ケア等に関連する施設基準届出 proxy である。

### 9.3 cap.homecare

> `cap.homecare` は、**訪問診療件数・看取り件数・在宅酸素実施件数を直接示さない**。  
> 在宅・訪問・居宅・往診等の届出 proxy である。

### 9.4 cap.rehab

> `cap.rehab` は、**実際のリハビリ実施件数やアウトカムを示さない**。  
> リハビリ関連施設基準の届出 proxy である。

### 9.5 cap.acute

> `cap.acute` は、**救急搬送受入件数や急性期医療の質を直接示さない**。  
> 救急・ICU/HCU/NICU・急性期関連施設基準の届出 proxy である。  
> 注: 「病床機能報告」の急性期病床数とも別概念。

### 9.6 Bridge supplyProxy との関係

Bridge Risk Model v1 (`docs/BRIDGE_V1_INTERPRETATION.md`) の `supply` フィールドで参照される `bedFuncKeys` (高度急性期/急性期/回復期/慢性期) は **病床機能報告データ (R6)** であり、本 capability_mapping とは **別データソース**。
